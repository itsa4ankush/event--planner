EventPilot AI — Hackathon MVP PRD

Multi-Agent Event Planning Assistant
Solo build · 12 hours · Submit target 10:00 AM CET (hard deadline 12:00 PM) · Started 9:15 PM CET

------------------------------------------------------------------------

0. TL;DR

EventPilot AI is a multi-agent system that turns a vague event idea (“FIFA World Cup viewing party, 50 people, my backyard”) into a structured plan: reasoned requirements, ranked vendor shortlist, a generated e-invite with RSVP tracking, a weather/legal/cost briefing, and a mock vendor-approval/payment flow — all on one dashboard.

Primary demo path (the thing that must work on stage):
Intake Q&A → Orchestrator reasons categories → Vendor Agent shortlists + ranks → user approves → Communication Agent makes one real Vapi call to confirm a quote → Invite Agent generates image + 10s video + RSVP page → guests RSVP → Cost/Weather/Legal Agent shows budget + forecast + guidance → user mock-approves an order/payment → Dashboard shows it all.

Intake input modes: the Orchestrator accepts both typed text and live voice (transcribed via Vapi speech-to-text) for the entire Q&A — date, location, guest count, theme, budget, etc. — using identical extraction logic regardless of channel.

Communication Agent now also handles a pitch-demo call: separate from any vendor outreach, the organizer can enter a phone number live on stage and Vapi places a real outbound call to it running a vendor-enquiry script — this showcases the conversational-calling capability without depending on a real vendor answering.

Why Level 2 and not Level 1 (single agent + tools): each specialist agent owns a distinct external system (search ranking, telephony, generative media, structured knowledge) with its own retry/fallback logic. A single agent juggling all four tool calls in one prompt loop is more failure-prone live on stage. The Orchestrator only ever talks to specialists, never calls Vapi/PixVerse/Cala directly.

Handoff contract: every specialist agent reads/writes through the database (Section 4), not direct agent-to-agent messaging. This is the simplest pattern to build solo in 12 hours and it doubles as your audit trail / dashboard data source for free.

3. Detailed PRD

3.1 Problem statement

Planning even a mid-size event (50-person watch party, corporate offsite, birthday) requires juggling requirement-gathering, vendor research, price comparison, outreach, invites, RSVPs, weather/logistics risk, and payments — usually across 6+ disconnected tools and phone calls. EventPilot AI consolidates this into one guided, agent-driven flow.

3.2 Target user

-   Primary (B2B): small companies/teams organizing corporate events, offsites, hackathons, launch parties — values speed, vendor comparison, and an auditable status board.
-   Secondary (B2C): individuals organizing private parties/watch parties — values guided Q&A and the invite/RSVP experience.

3.3 Core user story (reference scenario)

  “I want to host a FIFA World Cup viewing party in my backyard for 50 people.”

1.  Organizer enters this in plain language.
2.  Orchestrator infers required categories (food, drinks, snacks, seating/AV, decorations, merch/jerseys) and asks targeted follow-ups: which teams/tournament stage, indoor/outdoor, budget range, date/location.
3.  Vendor Research Agent searches Google/Yelp/Amazon-scoped sources for jersey/merch sellers and catering/drinks vendors near the location, ranks by review score + proximity + stated availability for the event date.
4.  Organizer reviews shortlist, approves a subset.
5.  Communication Agent places a real call (Vapi) to the top vendor to confirm price/availability; logs outcome to the Vendor record.
6.  Invite & RSVP Agent generates an image + 10s video invite and a shareable RSVP link/QR code.
7.  Guests RSVP with name, yes/no/maybe, contact, and comments (dietary, +1s).
8.  Cost/Weather/Legal Agent (Cala + weather API) returns: budget estimate, forecasted temperature/conditions for the date, a short “things to consider” list (permits/noise — superficial only), and optional alt-date or dress-code suggestions if weather looks bad.
9.  Organizer mock-approves the order; Payment/Order Agent shows a simulated Mollie checkout and marks vendor status approved.
10. Dashboard shows everything: KPI categories, vendor statuses, budget, RSVP count, invite preview.

3.4 Functional requirements by module

Intake/Orchestrator
- FR1: Accept free-text event description, extract event type, guest count, location, date.
- FR2: Infer relevant KPI categories (venue, food & drinks, decorations, photography, entertainment, merch) based on event type.
- FR3: Ask up to 3–5 targeted follow-up questions (event-type-specific, e.g. teams/tournament for sports events).
- FR4: Persist resulting EventBrief and create KPI category records with status not_started.
- FR4a: Support both typed text and live voice input (transcribed via Vapi speech-to-text) for the entire intake conversation; the same question/extraction logic applies regardless of input mode, with a simple toggle for the organizer to switch between them.

Vendor Research Agent
- FR5: For each approved category, run web search scoped to “Google/Yelp/Amazon” signals (ratings, review count, distance, stated availability).
- FR6: Rank vendors by a composite score (weights: rating, review count, availability match, proximity) — must support “100% reviews basis” mode the user described (rating-driven sort) as the default.
- FR7: Present shortlist (name, contact, rating, price estimate, source) for organizer approve/reject per vendor.

Communication Agent (Vapi)
- FR8: Prompt the organizer to enter a contact phone number; place a real outbound Vapi call to that number running a scripted vendor-enquiry conversation (price, availability, delivery timeline) — this is a pitch-demo mechanism, not a call to an actual vendor, and the UI should label it as such.
- FR9: Log call transcript/summary and extracted quote fields to VendorQuote.
- FR10: Other (non-demoed) vendors keep a simulated “outreach sent” status for demo continuity — no real calls are placed to them.

Invite & RSVP Agent (PixVerse)
- FR11: Generate one still image and one ≤10s video invite from event details (theme, date, location).
- FR12: Generate a unique RSVP page + QR code per event.
- FR12a: Provide Download buttons for the generated image and video files, and a shareable link (copy-to-clipboard / native share) so the organizer can send the invite directly to guests outside the app.
- FR13: Collect RSVP entries: name, response, contact, free-text comment (dietary/guests/etc.), timestamp.
- FR14: Show live RSVP count/list on dashboard.

Cost/Weather/Legal Intelligence Agent (Cala)
- FR15: Query Cala for relevant structured facts (e.g. typical labor/vendor cost ranges, general legal categories like permits/noise ordinances) tied to event type/location, presented as superficial, “consider this” guidance only — explicitly not legal advice.
- FR16: Query a weather API for forecasted conditions on the event date/location.
- FR17: Produce a budget estimate combining vendor quotes + Cala cost references.
- FR18: If forecast is poor (rain/extreme temp) for an outdoor event, surface alt-date suggestions and dress-code/expectation notes.

Payment/Order Agent (mock — visual only)
- FR19: Present a purely visual checkout mock-up summarizing approved vendor costs, styled with Mollie branding and a clear “Powered by Mollie” label — no real Mollie API call is made.
- FR20: On “approve,” mark vendor status approved (else rejected) and store a mock invoice record.

Dashboard
- FR21: Single view per event showing: KPI category progress, vendor list with status, budget estimate, RSVP list/count, invite preview, weather/legal panel.

3.5 Non-functional requirements

-   Demo must run end-to-end in under ~5 minutes without manual console intervention.
-   All external API failures must degrade to a clearly-labeled mock/fallback value rather than crashing the flow (critical given 12-hour build risk).
-   No PII beyond what’s needed for the demo (use placeholder/consented data for the live call).

3.6 Out of scope

See Section 1.3.

4. Database Schema

  Built inside Base44’s native data layer (or Postgres if you eject). Field types are indicative.

  ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  Table                               Key Fields
  ----------------------------------- ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  Organizer                           id, name, email, phone, company (nullable), created_at

  Event                               id, organizer_id, title, event_type (enum), date, time, location_text, lat, lng, guest_count, budget_total, status (draft/planning/confirmed/completed), created_at

  EventBriefAnswer                    id, event_id, question_key, answer_value (text/jsonb), input_mode (voice/text) — stores Q&A from intake

  KpiCategory                         id, event_id, name (venue/food_drinks/decor/photography/merch/entertainment), status (not_started/searching/shortlisted/approved/ordered)

  Vendor                              id, kpi_category_id, name, contact_email, contact_phone, source (google/yelp/amazon), rating, review_count, price_estimate, available_date, profile_url, status (shortlisted/contacted/quoted/approved/rejected)

  VendorQuote                         id, vendor_id, channel (voice_call/simulated_email), transcript_or_notes, quoted_price, delivery_timeline, outcome, created_at

  Invite                              id, event_id, image_url, video_url, qr_code_url, share_url, message_text, created_at

  Guest                               id, event_id, invite_id, name, contact, response (yes/no/maybe), comments, responded_at

  WeatherLegalReport                  id, event_id, forecast_temp, forecast_condition, alt_date_suggestions (jsonb), legal_notes (jsonb), dress_code_recommendation, generated_at

  Order                               id, event_id, vendor_id, amount, payment_status (mock_pending/mock_paid), invoice_ref, created_at
  ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Relationships: Organizer 1→N Event; Event 1→N KpiCategory; KpiCategory 1→N Vendor; Vendor 1→N VendorQuote; Event 1→N Guest (via Invite); Event 1→1 WeatherLegalReport; Event 1→N Order.

De-scope triggers (decide fast, don’t sunk-cost):
- If Vendor Research isn’t ranking real-looking data by 01:00 → fall back to a seeded sample dataset for 2 categories and move on.
- If the Vapi demo call isn’t connecting by 04:15 → fall back to the backup-recorded call clip. This risk is now lower than the original “cold-call a real vendor” plan since you control both ends of the call — but still test it once early in Phase 7, don’t leave it for game time.

- If PixVerse generation is slow/flaky → generate assets early and cache; never regenerate live during the demo.
